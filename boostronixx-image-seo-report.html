Critical Finding: The SVG Placeholder Trap

🚨 Every single image on the site is a base64-encoded SVG placeholder — not the real image. This is a Perfmatters/Smush-style JS lazy loader. Googlebot sees ONLY SVG placeholders, not the actual photos. This is the single biggest image SEO issue on the site.

🔴 CRITICAL — All Images Replaced with SVG Lazy-Load Placeholders

Every <img> tag's src contains a base64-encoded SVG rectangle (grey box) instead of the real image URL. The real URL is stored in a data-src or similar attribute, loaded by JavaScript after page paint. Googlebot renders images inconsistently in JS-lazy-load setups, and with the current SVG placeholder pattern, it is almost certain Googlebot is NOT seeing any real images on the site.

This explains the indexation crisis (only 3 pages indexed) — Google cannot evaluate image content, OG images, or hero visuals. It hurts E-E-A-T signals and makes the site appear visually empty to crawlers that don't execute advanced JavaScript.

Root cause

A WordPress plugin (most likely Perfmatters, WP Rocket, or Smush) is replacing all src attributes with inline SVG placeholders and storing real URLs in data-perfmatters-src, data-lazy, or similar data attributes.

Fix
Step 1: In WordPress Admin → go to the lazy-loading plugin settings
Step 2: DISABLE lazy loading for:
  - The hero / above-the-fold image
  - All images in the first viewport (top ~600px of any page)
  - Logo images in header
Step 3: Add fetchpriority="high" to hero image
Step 4: Keep lazy loading ONLY for images below the fold
Step 5: Test with Google's Rich Results Test to verify images now load

Lazy Loading Analysis

🚨 Lazy loader method: js-generic (data-src / base64 SVG placeholder pattern). The plugin is replacing ALL image src attributes with inline SVG placeholders BEFORE the browser and Googlebot can request the real images. This breaks Googlebot image indexing.

🔴 Hero image is being lazy-loaded — Direct LCP penalty

The homepage hero image (692×443) is the LCP element. It is replaced by a base64 SVG placeholder, meaning the browser must wait for JavaScript to execute before starting to download the real image. This adds 500ms–2000ms to LCP on slow mobile connections. Google PageSpeed will flag this.

Fix for hero image
<!-- WRONG (current) -->
<img src="data:image/svg+xml;base64,..." alt="Creative Branding Agency for Ambitious Businesses">

<!-- CORRECT -->
<img
  src="https://boostronixx.s3.ap-south-1.amazonaws.com/.../hero-image.webp"
  alt="Creative Branding Agency for Ambitious Businesses — BoostronixX Jaipur"
  width="692"
  height="443"
  fetchpriority="high"
  decoding="auto"
>
<!-- NO loading="lazy" on hero! -->

⚠️ Logo images also lazy-loaded — CLS and UX risk

Both the desktop (189×38) and mobile/sticky (1567×394) logo variants are replaced with SVG placeholders. Logos should NEVER be lazy-loaded — they appear in the first byte of the viewport and are required before page is usable.

Fix in your lazy-loading plugin
In WordPress plugin settings (Perfmatters / WP Rocket / Smush):
→ Add the logo image selector to the "excluded images" list
→ Typical selector: .site-logo img, header img, .navbar-brand img
→ OR: Add class="skip-lazy" to logo <img> tags in your theme and
  exclude that class in the plugin settings

⚠️ 1×1 GIF Tracking Pixels Used as Icons

Four instances of data:image/gif;base64,R0lGODdhAQABAPAAAMPDwwAAACwAAAAAAQABAAACAkQBADs= (a 1×1 transparent GIF) are used as arrow/social icons. These are legacy tracking pixel patterns that should be replaced with inline SVG or proper icon fonts.

Fix
<!-- Replace 1×1 GIF arrows with inline SVG -->
<svg width="19" height="74" aria-hidden="true" focusable="false" viewBox="0 0 19 74">
  <!-- your arrow path here -->
</svg>

Correct Lazy Loading Pattern

Best practice: Split above-fold vs below-fold
<!-- ABOVE THE FOLD: Hero, logo, first visible content -->
<img

⚠️ No WebP Conversion for Body Images

Only the favicon uses WebP. All body images appear to be JPEG or PNG served from AWS S3. WebP offers 25–35% smaller file sizes vs JPEG at equivalent quality, directly improving LCP and page load times.

Batch conversion with WordPress plugin
Option 1 (Recommended — WordPress): Install ShortPixel or Imagify plugin
  → Bulk convert all uploaded images to WebP
  → Enable "Serve WebP images" option
  → WordPress will automatically use <picture> tag with WebP + JPEG fallback

Option 2 (Manual — command line):
  # Install cwebp
  sudo apt install webp

  # Convert single image
  cwebp -q 82 hero-image.jpg -o hero-image.webp

  # Batch convert all JPEGs in a folder
  for f in *.jpg; do cwebp -q 82 "$f" -o "${f%.jpg}.webp"; done

Option 3 (S3 + CloudFront): Configure CloudFront to serve WebP via
  Content-Type negotiation based on Accept header

Recommended picture element with format fallback

<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img
    src="image.jpg"
    alt="SEO services for Jaipur businesses — BoostronixX"
    width="446"

    decoding="async"

CLS Prevention — Dimensions Status

The SVG placeholders encode the correct width/height of each image (readable from the base64 viewBox), which means the browser CAN reserve the correct space before the real image loads. This is the one accidental benefit of the current lazy-loading setup — CLS is likely NOT a major issue. However, this should be verified with Chrome DevTools after fixing the lazy-loading problem.

✅ SVG placeholders contain correct dimensions — CLS impact is low. When switching to native lazy loading, ensure width + height attributes are present on all img tags.
